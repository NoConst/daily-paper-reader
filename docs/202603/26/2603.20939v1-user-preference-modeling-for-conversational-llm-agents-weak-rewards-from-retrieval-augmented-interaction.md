## 摘要
大语言模型越来越多地被用作个人助手，但大多数模型缺乏持久的用户模型，导致用户不得不在不同会话中重复陈述偏好。我们提出了向量自适应检索评分（Vector-Adapted Retrieval Scoring, VARS），这是一个与流水线无关、冻结骨干网络的框架。该框架在共享偏好空间中利用长期和短期向量来表示每个用户，并使用这些向量来偏置结构化偏好记忆上的检索评分。这些向量根据用户反馈的弱标量奖励进行在线更新，从而在无需针对每个用户进行微调的情况下实现个性化。我们在 MultiSessionCollab 上进行了评估，这是一个包含丰富用户偏好配置文件的在线多会话协作基准测试，涵盖了数学和代码任务。在冻结骨干网络的情况下，用户感知检索的主要优势在于提高交互效率，而非大幅提升原始任务准确率：我们的完整 VARS 智能体实现了最强的综合性能，在任务成功率上与强大的 Reflection 基准相当，并降低了超时率和用户工作量。学习到的长期向量还与跨用户偏好重叠相一致，而短期向量则捕捉了特定会话的适应性，这支持了双向量设计的可解释性。代码、模型和数据可在 https://github.com/YurenHao0426/VARS 获取。

## Abstract
Large language models are increasingly used as personal assistants, yet most lack a persistent user model, forcing users to repeatedly restate preferences across sessions. We propose Vector-Adapted Retrieval Scoring (VARS), a pipeline-agnostic, frozen-backbone framework that represents each user with long-term and short-term vectors in a shared preference space and uses these vectors to bias retrieval scoring over structured preference memory. The vectors are updated online from weak scalar rewards from users' feedback, enabling personalization without per-user fine-tuning. We evaluate on \textsc{MultiSessionCollab}, an online multi-session collaboration benchmark with rich user preference profiles, across math and code tasks. Under frozen backbones, the main benefit of user-aware retrieval is improved interaction efficiency rather than large gains in raw task accuracy: our full VARS agent achieves the strongest overall performance, matches a strong Reflection baseline in task success, and reduces timeout rate and user effort. The learned long-term vectors also align with cross-user preference overlap, while short-term vectors capture session-specific adaptation, supporting the interpretability of the dual-vector design. Code, model, and data are available at https://github.com/YurenHao0426/VARS.

### 1. 核心问题与研究动机
*   **核心问题**：当前大语言模型（LLM）作为个人助手时，普遍缺乏**持久的用户模型**。这导致用户在不同会话（Session）中必须重复陈述其特定偏好（如代码风格、数学推导详略等），降低了长期协作的效率。
*   **研究背景**：现有的长短期记忆方案多依赖于简单的 RAG（检索增强生成）或会话总结（Reflection），但它们通常仅基于查询相似度进行检索，无法根据用户与系统的持续交互动态调整检索优先级，也难以区分用户的长期稳定偏好与短期临时需求。

### 2. 方法论（VARS 框架）
VARS（Vector-Adapted Retrieval Scoring）是一个**无需微调骨干模型**、基于冻结参数（Frozen-backbone）的个性化框架。其核心组件包括：
*   **偏好提取（Preference Extraction）**：使用一个轻量级微调模型（Qwen3-0.6B）将对话实时转换为结构化的 `(condition, action)` JSON 元组（如：当解决代数题时，显示每一步推导）。
*   **双向量用户状态（Dual-Vector User State）**：
    *   **长期向量 ($z^{(L)}$)**：跨会话持久存在，捕捉用户稳定的风格偏好。
    *   **短期向量 ($z^{(S)}$)**：在会话内更新并随时间衰减，捕捉当前任务的特定上下文。
*   **用户感知检索评分**：在检索偏好记忆时，最终评分 = 基础重排序分数 + 用户向量与记忆项向量的点积。这种“低秩残差评分层”使得检索结果向用户历史偏好倾斜。
*   **在线弱奖励更新**：
    *   **弱奖励信号**：通过启发式规则分析用户下一轮的反馈（如关键词“谢谢”、“错误”、“重做”等）生成标量奖励 $\hat{r}_t$。
    *   **更新算法**：采用 **REINFORCE 风格**的策略梯度更新用户向量。引入“检索归因门控（Gating）”来判断奖励是由检索质量还是模型生成能力引起的，从而减少更新噪声。

### 3. 实验设计
*   **基准测试**：使用 **MultiSessionCollab**，这是一个模拟长期协作的基准，包含 60 个具有丰富偏好配置的用户画像。
*   **任务场景**：涵盖 Math-hard（复杂数学）、Math-500（通用数学）和 BigCodeBench（代码生成）三个领域。
*   **对比方法**：
    *   **Vanilla**：无记忆。
    *   **Contextual**：全历史拼接。
    *   **All-memory**：注入所有提取的偏好。
    *   **Reflection**：基于会话总结的强基准。
    *   **RAG**：标准检索重排序（无用户向量）。
*   **实验规模**：每个方法评估 60 个画像 × 60 个会话，总计 3,600 个会话。

### 4. 资源与算力
*   **训练算力**：偏好提取模型（Mext）在 **H200 GPU** 上进行全参数微调，训练数据集包含约 56.4 万条样本。

*   **向量维度**：用户向量和记忆项向量通过 PCA 降维至 256 维，以降低计算开销。

### 5. 实验数量与充分性
*   **实验规模**：论文进行了大规模的自动化评估（3,600 个会话/方法），样本量充足。
*   **消融实验**：针对双向量设计（仅长期、仅短期、全量）进行了对比，验证了两者在减少超时和提升成功率上的不同分工。
*   **敏感性分析**：对奖励信号的关键词噪声、归因门控的阈值进行了鲁棒性测试，证明了框架在弱信号下的稳定性。
*   **客观性**：通过与强基准（Reflection）的配对 t 检验（Paired t-test）来验证显著性，实验设计较为严谨。

### 6. 主要结论与发现
*   **效率胜过准确率**：VARS 的主要贡献在于**提高交互效率**。虽然在原始任务成功率上与 Reflection 相当，但显著降低了**超时率**和**用户工作量（Token 数）**。
*   **缓解偏好过载**：相比于盲目注入所有记忆，VARS 能通过用户向量过滤掉不相关的偏好，避免智能体因指令过多而产生困惑。
*   **向量可解释性**：学习到的长期向量在空间上的距离与用户真实的偏好重叠度呈正相关，证明向量确实捕捉到了用户身份特征。

### 7. 优点
*   **轻量且即插即用**：无需对大型骨干模型进行昂贵的每用户微调（Per-user fine-tuning），仅需维护极小维度的向量。
*   **双尺度建模**：成功分离了用户的“本性”（长期）与“现状”（短期），提高了系统在复杂多变对话中的适应力。
*   **闭环学习**：通过弱反馈实现持续在线进化，无需显式的用户标注。

### 8. 不足与局限
*   **模拟器偏差**：实验完全依赖 LLM 模拟用户，可能无法完全反映真实人类的耐心、表达模糊性或偏好转变。
*   **奖励信号单一**：目前主要依赖关键词匹配，对于微妙的、隐含的负面反馈捕捉能力有限。
*   **隐私风险**：持久化的用户偏好建模和结构化存储可能涉及隐私泄露风险，文中对此仅做了简要提及。
*   **任务覆盖**：虽然涵盖了数学和代码，但对于开放式创意写作或情感支持等主观性更强的任务尚未验证。
