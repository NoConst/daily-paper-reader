## 摘要
大语言模型（LLM）智能体在长上下文交互中经常面临挑战。随着智能体积累更多的交互历史，滑动窗口和提示词压缩等上下文管理方法可能会遗漏后续步骤所依赖的早期结构化信息。最近的基于检索的记忆系统虽然能提取相关内容，但仍忽视了多步推理所需的因果和逻辑结构。我们提出了 ContextWeaver，这是一个选择性且具有依赖结构化的记忆框架，它将智能体的交互轨迹组织成推理步骤图，并为未来的行动选择相关的上下文。与之前的上下文管理方法不同，ContextWeaver 支持：（1）基于依赖的构建和遍历，将每一步与其所依赖的早期步骤联系起来；（2）紧凑的依赖摘要，将从根到步骤的推理路径压缩为可重用的单元；（3）一个结合了执行反馈的轻量级验证层。在 SWE-Bench Verified 和 Lite 基准测试中，ContextWeaver 在 pass@1 指标上优于滑动窗口基准，同时减少了推理步骤和 Token 使用量。我们的观察表明，建模逻辑依赖关系为使用工具的 LLM 智能体提供了一种稳定且可扩展的记忆机制。

## Abstract
Large language model (LLM) agents often struggle in long-context interactions. As the agent accumulates more interaction history, context management approaches such as sliding window and prompt compression may omit earlier structured information that later steps rely on. Recent retrieval-based memory systems surface relevant content but still overlook the causal and logical structure needed for multi-step reasoning. We introduce ContextWeaver, a selective and dependency-structured memory framework that organizes an agent's interaction trace into a graph of reasoning steps and selects the relevant context for future actions. Unlike prior context management approaches, ContextWeaver supports: (1) dependency-based construction and traversal that link each step to the earlier steps it relies on; (2) compact dependency summarization that condenses root-to-step reasoning paths into reusable units; and (3) a lightweight validation layer that incorporates execution feedback. On the SWE-Bench Verified and Lite benchmarks, ContextWeaver improves performance over a sliding-window baseline in pass@1, while reducing reasoning steps and token usage. Our observations suggest that modeling logical dependencies provides a stable and scalable memory mechanism for LLM agents that use tools.

### 2. 方法论：核心思想与关键技术
ContextWeaver 提出了一种**选择性且具有依赖结构化**的记忆框架，其核心是将交互轨迹维护为一个**有向无环图（DAG）**。

*   **核心思想**：不按时间顺序或语义相似性选择上下文，而是按“逻辑依赖”选择。
*   **关键技术组件**：
    1.  **依赖感知构建（Dependency-Aware Construction）**：使用 LLM 作为“逻辑依赖分析器”，为每个新步骤（节点）识别其父节点（即该步骤所依赖的先前操作），构建 DAG。
    2.  **依赖摘要器（Dependency Summarizer）**：增量式地为推理路径生成紧凑摘要，记录从根节点到当前节点的逻辑流，避免重复处理完整历史。
    3.  **验证与测试层（Validation Layer）**：记录工具执行结果（如测试通过/失败）。失败或被取代（Superseded）的节点在后续构建中会被跳过，确保智能体只基于可靠的证据进行推理。
    4.  **上下文编织（Context Weaving）**：根据当前步骤的祖先节点（Ancestry Set）构建最终提示词。对于祖先节点保留完整的“思考-行动-观察”三元组，对于非祖先节点则压缩其观察结果。

*   **模型依赖性**：该方法的有效性取决于“依赖分析器”的质量。在混合设置中，高质量的记忆图能显著提升较弱模型的表现。

### 7. 优点
*   **逻辑严密性**：通过显式建模逻辑依赖，解决了智能体在长交互中“迷失”或“复读”的问题。
*   **动态验证**：引入验证层，利用执行反馈（测试结果）动态剪枝，防止错误信息在记忆中传播。
*   **可解释性**：生成的依赖图和路径摘要为智能体的决策过程提供了清晰的溯源。
*   **即插即用**：作为一个记忆管理层，它可以集成到现有的各种智能体框架（如 SWE-agent, OpenHands）中。

### 8. 不足与局限
*   **计算开销**：虽然节省了 Agent 侧的 Token，但构建和维护依赖图（需要额外的 LLM 调用进行分析和摘要）会带来额外的 API 成本和延迟。
*   **任务局限性**：目前主要在具有明确执行反馈（测试驱动）的软件工程任务中验证，在缺乏自动验证信号的开放域任务中效果尚待观察。
*   **依赖分析器的瓶颈**：如果负责构建图的模型能力不足，可能会产生错误的依赖链接，导致智能体获取错误的上下文。
*   **顺序任务的冗余**：在极其简单的线性任务中，相比简单的滑动窗口，该方法可能会引入不必要的计算复杂度。
