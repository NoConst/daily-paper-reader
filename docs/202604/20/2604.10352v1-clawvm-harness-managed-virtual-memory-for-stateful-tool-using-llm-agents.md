## 摘要
有状态的工具调用型 LLM 智能体将上下文窗口视为工作内存，然而现有的智能体框架对驻留性和持久性的管理仅是尽力而为，导致了频繁发生的故障：压缩后的状态丢失、重置时跳过刷新以及破坏性回写。我们提出了 ClawVM，这是一个虚拟内存层，它将状态管理为具有最小保真度不变式的有类型页，在 Token 预算下支持多分辨率表示，并在每个生命周期边界进行经验证的回写。由于框架已经负责组装提示词、协调工具并观察生命周期事件，它是天然的执行点；将契约置于此处使驻留性和持久性变得确定且可审计。在合成工作负载、12 个真实会话追踪和对抗性压力测试中，ClawVM 在最小保真度集合符合 Token 预算的情况下消除了所有策略可控故障（经离线真值确认），且每轮增加的策略引擎中位开销小于 50 微秒。

## Abstract
Stateful tool-using LLM agents treat the context window as working memory, yet today's agent harnesses manage residency and durability as best-effort, causing recurring failures: lost state after compaction, bypassed flushes on reset, and destructive writeback. We present \textsc{ClawVM}, a virtual memory layer that manages state as typed pages with minimum-fidelity invariants, multi-resolution representations under a token budget, and validated writeback at every lifecycle boundary. Because the harness already assembles prompts, mediates tools, and observes lifecycle events, it is the natural enforcement point; placing the contract there makes residency and durability deterministic and auditable. Across synthetic workloads, 12 real-session traces, and adversarial stress tests, \textsc{ClawVM} eliminates all policy-controllable faults whenever the minimum-fidelity set fits within the token budget, confirmed by an offline oracle, and adds median <50 microseconds of policy-engine overhead per turn.

### 论文总结：ClawVM - 面向有状态工具调用型 LLM 智能体的框架管理虚拟内存

#### 1. 核心问题与整体含义（研究动机和背景）
随着 LLM 智能体（如 OpenClaw、Claude Code）在长会话中处理复杂任务（如邮件管理、代码编写），它们需要跨越多个上下文窗口管理持久状态。然而，现有的智能体框架（Harness）在管理上下文内存时通常采用“尽力而为”的启发式策略，导致了三大类故障：
*   **驻留性故障**：在上下文压缩（Compaction）时丢失关键指令或计划步骤。
*   **持久性故障**：在会话重置时跳过内存刷新，或发生破坏性的覆盖回写。
*   **可观测性故障**：无法区分“未找到匹配”与“后端错误”。
论文的核心动机是将操作系统中的**虚拟内存**概念引入智能体框架，通过建立确定性的内存契约，确保关键状态在生命周期边界（如压缩、重置）不丢失。

#### 2. 方法论：核心思想与关键技术
ClawVM 将智能体框架视为“内核”，将上下文窗口视为“物理内存”，将外部存储视为“磁盘”。其核心技术包括：
*   **类型化页面（Typed Pages）**：将状态划分为引导/策略、约束、计划、偏好、证据和对话片段等类型，每种类型都有最小保真度要求。
*   **多分辨率表示（Multi-resolution Representations）**：页面可以以四种级别存在：完整（Full）、压缩（Compressed）、结构化（Structured）和指针（Pointer）。在 Token 预算紧张时，系统会优雅降级而非直接丢弃。
*   **确定性选择策略**：采用两阶段贪心算法。第一阶段安装所有硬固定（Hard-pinned）页面；第二阶段根据“边际效用/Token 成本”进行升级。
*   **验证回写协议（Validated Writeback）**：分为结构化暂存、确定性验证（检查模式、来源、非破坏性语义）和作用域提交三个阶段，防止错误覆盖。
*   **故障模型**：定义了可观测的故障类型（如 Refetch、Flush-miss、Bootstrap fault），使内存管理决策可审计、可重放。

#### 5. 实验数量与充分性
*   **实验规模**：涵盖了 24 种配置（4 种负载 × 6 种 Token 预算）、12 个真实长会话追踪、30 个任务级重放以及 6 个生命周期回归场景。
*   **充分性与公平性**：实验设计非常充分。通过消融实验验证了各组件（如指针解析、自动固定）的必要性；通过权重敏感度分析证明了策略的鲁棒性；引入 Oracle 对比证明了该策略已达到理论最优。对比基准（Comp-Hybrid）参考了主流框架的文档和社区讨论，具有代表性。

#### 6. 主要结论与发现
*   **故障消除**：在 Token 预算允许的情况下，ClawVM 彻底消除了所有策略可控故障（从基准的平均 67.8 个降至 0）。
*   **稳定性提升**：相比从业者配置的基准，抖动指数（Thrash Index）降低了 11.4%。
*   **鲁棒性**：即使在极端的 Token 预算压力或频繁重置的对抗性场景下，ClawVM 依然能保持状态的完整性。
*   **确定性**：安全性的提升源于框架层的强制执行（Enforcement），而非对启发式参数的微调。

#### 7. 优点
*   **系统性强**：首次将成熟的虚拟内存管理机制完整引入 LLM 智能体领域，解决了状态管理的确定性问题。
*   **开销极低**：微秒级的延迟增加对于秒级的 LLM 推理几乎可以忽略不计。
*   **可审计性**：通过故障模型和决策日志，使智能体的“失忆”或“错误”变得可诊断、可重现。
*   **兼容性**：作为框架层插件，不依赖特定的模型或检索后端。

#### 8. 不足与局限
*   **语义正确性**：ClawVM 只能保证状态“被正确保存和加载”，但无法验证模型生成的更新内容在语义上是否真实（即无法解决幻觉问题）。
*   **工具模拟限制**：重放实验基于工具签名的模拟，对于非确定性服务的在线效应模拟可能存在偏差。
*   **应用范围**：目前主要在重放负载和单会话任务中验证，多智能体协作及更复杂的跨会话长期部署仍需进一步研究。
