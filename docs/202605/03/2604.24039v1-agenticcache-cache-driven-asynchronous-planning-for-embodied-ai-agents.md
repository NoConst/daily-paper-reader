## 摘要
具身智能体（Embodied AI agents）越来越多地依赖大语言模型（LLMs）进行规划，但每步调用 LLM 会带来严重的延迟和成本。在本文中，我们展示了具身任务表现出极强的规划局部性（plan locality），即下一个规划在很大程度上可以根据当前规划进行预测。基于此，我们提出了 AgenticCache，这是一个通过重用缓存规划来避免每步调用 LLM 的规划框架。在 AgenticCache 中，每个智能体查询频繁规划转换的运行时缓存，而后台的缓存更新器（Cache Updater）则异步调用 LLM 以验证和细化缓存条目。在四个多智能体具身基准测试中，AgenticCache 在 12 种配置（4 个基准测试 x 3 个模型）下平均将任务成功率提高了 22%，模拟延迟降低了 65%，Token 使用量减少了 50%。因此，基于缓存的规划重用为实现低延迟、低成本的具身智能体提供了一条切实可行的路径。代码可在 https://github.com/hojoonleokim/MLSys26_AgenticCache 获取。

## Abstract
Embodied AI agents increasingly rely on large language models (LLMs) for planning, yet per-step LLM calls impose severe latency and cost. In this paper, we show that embodied tasks exhibit strong plan locality, where the next plan is largely predictable from the current one. Building on this, we introduce AgenticCache, a planning framework that reuses cached plans to avoid per-step LLM calls. In AgenticCache, each agent queries a runtime cache of frequent plan transitions, while a background Cache Updater asynchronously calls the LLM to validate and refine cached entries. Across four multi-agent embodied benchmarks, AgenticCache improves task success rate by 22% on average across 12 configurations (4 benchmarks x 3 models), reduces simulation latency by 65%, and lowers token usage by 50%. Cache-based plan reuse thus offers a practical path to low-latency, low-cost embodied agents. Code is available at https://github.com/hojoonleokim/MLSys26_AgenticCache.

*   **研究动机**：作者发现具身任务具有强烈的**“规划局部性”（Plan Locality）**，即智能体的下一步规划往往可以从当前状态和历史模式中预测（例如：抓取物体后，下一步通常是移动到目标点）。受计算机体系结构中缓存机制的启发，作者提出通过缓存频繁的规划转换来减少对LLM的实时依赖。

### 2. 方法论
*   **核心思想**：建立一个运行时缓存来存储“规划-到-规划”的转换模式，由智能体优先查询缓存进行快速决策，同时由后台进程异步调用LLM来验证和修正缓存内容。
*   **关键技术细节**：
    *   **AgenticCache 结构**：采用 2-Gram 模式存储转换 $\langle P_i \to P_j \rangle$，并附带元数据（如步骤数、持有物品数等状态范围）。
    *   **评分机制**：通过公式 $S(P_i \to P_j) = C \cdot I$ 选择最佳候选规划。其中 $C$ 是局部转换频率，$I$ 是全局重要性（由LLM确认的比例）。
    *   **异步更新器（Cache Updater）**：在后台运行，不阻塞智能体行动。它负责：
        1.  **确认（Confirmation）**：若LLM建议与缓存一致，增强该缓存条目。
        2.  **修正（Correction）**：若不一致，立即替换当前正在执行的错误规划（Plan Replacement），并更新缓存。
    *   **查询抑制**：当缓存与LLM达成共识或正在执行修正规划时，暂停冗余的LLM查询以节省成本。
    *   **预填充（Warm-start）**：可选步骤，利用少量离线成功轨迹初始化缓存，解决冷启动问题。
