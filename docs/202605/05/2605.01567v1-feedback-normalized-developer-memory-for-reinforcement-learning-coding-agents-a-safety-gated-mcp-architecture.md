## 摘要
大语言模型（LLM）编程智能体越来越多地在长软件工程周期中处理代码库、终端、测试和执行轨迹。持久化记忆非常有用，但静态向量存储或通用检索增强生成（RAG）对于强化学习（RL）代码开发而言是不够的，因为微小的细节可能会改变 Bellman 目标、终止掩码、梯度流或验证声明。本文提出了 RL 开发者记忆（RL Developer Memory），这是一种面向 RL 编程智能体的本地优先、原生支持模型上下文协议（MCP）的开发者记忆架构。它将记忆选择视为一个记录在案的上下文决策过程：issue_match 对候选项进行排序并记录遥测数据，issue_feedback 将原始标签映射到有界奖励，issue_record_resolution 将经过验证的解决方案链接到早期的检索事件。系统部署了一个确定性排序器，而上下文老虎机（contextual-bandit）残差策略以影子模式运行，且仅能通过保守的离线策略评估（OPE）门控影响金丝雀行为。RL/控制记忆需要理论到代码的元数据和经过评审门控的治理。该系统在一个包含 200 个确定性案例的基准测试上进行了评估，涵盖了 RL 算法漏洞、硬负样本、评审门控的 RL/控制案例以及低风险故障。在同提交对比中，确定性控制和完整的影子/OPE 模式均实现了 80.0% 的预期决策准确率和 100.0% 的硬负样本抑制；完整配置增加了学习遥测数据而非准确率提升。静态验证通过了 11/11 项检查；动态集成通过了 10/10 个案例。证据报告了局限性：不支持主动学习策略部署和官方客户端 MCP 互操作性，实时完整配置的延迟有所增加，且仍存在 40 个残余的非 RL 故障。本研究的贡献在于提出了一种具有明确声明边界的可审计记忆控制架构，而非通用的编程智能体改进声明。

## Abstract
Large language model (LLM) coding agents increasingly operate over repositories, terminals, tests, and execution traces across long software-engineering episodes. Persistent memory is useful, but static vector stores or generic retrieval-augmented generation (RAG) are insufficient for reinforcement-learning (RL) code development, where small details can alter Bellman targets, terminal masks, gradient flow, or validation claims. This paper presents RL Developer Memory, a local-first, Model Context Protocol (MCP)-native developer-memory architecture for RL coding agents. It treats memory selection as a logged contextual decision process: issue_match ranks candidates and records telemetry, issue_feedback maps raw labels to bounded rewards, and issue_record_resolution links verified resolutions to earlier retrieval events. A deterministic ranker remains deployed, while a contextual-bandit residual policy runs in shadow mode and can affect canary behavior only through conservative off-policy-evaluation (OPE) gates. RL/control memories require theory-to-code metadata and review-gated governance. The system is evaluated on a deterministic 200-case benchmark with RL algorithm bugs, hard negatives, review-gated RL/control cases, and low-risk failures. In the same-commit comparison, deterministic control and full shadow/OPE both achieve 80.0% expected-decision accuracy and 100.0% hard-negative suppression; the full configuration adds learning telemetry rather than accuracy gain. Static validation passed 11/11 checks; dynamic integration passed 10/10 cases. The evidence reports limits: active learned-policy deployment and official-client MCP interoperability are unsupported, live full-configuration latency regresses, and 40 residual non-RL failures remain. The contribution is an auditable memory-control architecture with explicit claim boundaries, not a universal coding-agent improvement claim.

#### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）编程智能体从简单的代码补全转向复杂的软件工程任务（如处理代码库、运行测试、修复 Bug），**持久化记忆**变得至关重要。然而，传统的检索增强生成（RAG）或静态向量存储在处理**强化学习（RL）算法代码**时存在严重局限。RL 代码具有极高的数学敏感性：微小的实现差异（如 Bellman 目标的计算、梯度流边界、终止掩码或目标网络更新逻辑）都会导致算法失效。

本文的研究动机在于构建一种能够精准捕捉 RL 算法细节、具备安全门控机制且可审计的开发者记忆架构，以解决通用 RAG 在处理高风险、高精度编程任务时的不足。

#### 2. 方法论：核心思想与关键技术
论文提出了 **RL Developer Memory**，这是一种基于模型上下文协议（MCP）的本地优先架构。其核心思想是将记忆检索视为一个**受控的上下文决策过程**：

*   **上下文决策建模**：系统不仅检索相似代码，还将每次检索记录为决策事件。
*   **反馈归一化（Feedback Normalization）**：通过 `issue_feedback` 工具将开发者零散、非规范的反馈（如“有用”、“错误”）映射为有界的规范奖励值（如 -1.0 到 1.0），用于训练。
*   **延迟奖励链接（Delayed Reward Linking）**：通过 `issue_record_resolution` 将后期验证成功的修复方案与早期的检索事件关联，解决强化学习中的信用分配（Credit Assignment）问题。
*   **双路径控制架构**：
    *   **确定性路径（控制组）**：当前部署的策略，基于加权排序。
    *   **影子模式（实验组）**：运行上下文老虎机（Contextual-bandit）残差策略，记录倾向性但不直接干预生产，仅在通过**离线策略评估（OPE）**门控后才允许进行金丝雀发布。
*   **理论到代码的追溯（Theory-to-Code Traceability）**：要求记忆条目携带元数据，将数学公式（如 TD 误差）与代码锚点（文件、行号）绑定，确保修复符合算法理论。

#### 3. 实验设计
*   **数据集/场景**：构建了一个包含 **200 个确定性案例** 的 RL 开发者记忆基准测试。
*   **Benchmark 组成**：
    *   涵盖多种 RL 算法家族（DQN, PPO, SAC, TD3, GAE, A2C, V-trace）。
    *   包含 80 个“硬负样本”（Hard Negatives），即看起来相似但逻辑错误的干扰项。
    *   包含 40 个非 RL 的通用开发者记忆故障（如路径处理、项目范围问题）。
*   **对比方法**：主要对比了“确定性控制路径”与“完整影子/OPE 配置”在相同提交下的表现。

#### 5. 实验数量与充分性
*   **实验规模**：进行了 200 组基准测试案例，涵盖静态验证（11 项）、动态集成（10 项）以及离线/在线影子模式运行。
*   **充分性与客观性**：实验设计较为严谨，采用了“同提交受控对比”（Same-commit controlled comparison），并明确区分了“方法基线”与“补丁重放审计”。作者诚实地指出，虽然补丁重放显示了正向增量，但受控对比显示完整配置在准确率上并未直接超越确定性基线，其主要贡献在于**遥测数据的收集和可审计性**。

#### 6. 主要结论与发现
*   **准确性**：系统在 200 个案例中实现了 **80.0% 的决策准确率** 和 **100.0% 的硬负样本抑制**。
*   **遥测价值**：完整配置成功记录了 60.5% 的上下文统计更新率，证明了影子模式在不干扰生产的情况下收集学习数据的可行性。
*   **性能代价**：实时配置下的 p95 延迟有所增加（从约 447ms 增加到 852ms），主要源于指标记录和 OPE 报告的开销。
*   **失败分析**：残余的 40 个失败案例全部集中在非 RL 领域（如 Windows 挂载路径处理、项目范围缺失），说明系统在通用编程环境的鲁棒性仍有待提高。

#### 7. 优点
*   **安全性高**：引入 OPE 门控和影子模式，防止未经验证的 RL 策略直接干预代码生成。
*   **领域针对性强**：专门为 RL 算法设计了“理论到代码”的映射，解决了通用 RAG 无法处理的算法逻辑一致性问题。
*   **架构标准化**：原生支持 MCP 协议，方便集成到现代 AI 编程工具链中。
*   **审计友好**：所有反馈和决策路径均可追溯，符合软件工程的合规性要求。

#### 8. 不足与局限
*   **延迟回归**：全量配置下的系统延迟显著增加，可能影响开发者体验。
*   **消融实验缺失**：未完成 V0-V4 的逐项消融实验（如单独验证反馈归一化或延迟链接的贡献）。
*   **通用性受限**：在处理非 RL 的常规编程任务（如路径、别名兼容性）时表现不佳。
*   **部署限制**：目前尚不支持主动学习策略的完全自动化部署，且与官方 MCP 客户端的互操作性尚未完全验证。
