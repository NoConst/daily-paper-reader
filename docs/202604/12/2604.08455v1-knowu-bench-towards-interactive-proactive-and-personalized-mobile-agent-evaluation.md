## 摘要
能够推断用户偏好并校准主动协助的个性化移动智能体，作为日常数字助手具有巨大潜力，但现有的基准测试未能捕捉到实现这一目标所需的能力。先前的工作主要评估从静态历史记录中恢复偏好，或从固定上下文中预测意图。这些工作既没有测试智能体是否能通过交互引导出缺失的偏好，也没有测试其在实时 GUI 环境中能否决定何时干预、寻求许可或保持沉默。我们推出了 KnowU-Bench，这是一个基于可复现 Android 模拟环境的个性化移动智能体在线基准测试，涵盖了 42 个通用 GUI 任务、86 个个性化任务和 64 个主动式任务。与以往将用户偏好视为静态上下文的工作不同，KnowU-Bench 对智能体隐藏了用户画像，仅公开行为日志，从而强制进行真正的偏好推断而非简单的上下文查找。为了支持多轮偏好引导，它实例化了一个基于结构化画像的大语言模型（LLM）驱动的用户模拟器，从而实现真实的澄清对话和主动许可处理。除了个性化之外，KnowU-Bench 还对完整的主动决策链提供了全面评估，包括基于环境的 GUI 执行、许可协商以及被拒绝后的克制，评估采用结合了基于规则的验证与 LLM-as-a-Judge 评分的混合协议。我们的实验揭示了显著的性能下降：即使是像 Claude Sonnet 4.6 这样的前沿模型，在擅长执行明确任务的情况下，面对需要用户偏好推断或干预校准的模糊指令时，其表现也会降至 50% 以下。核心瓶颈不在于 GUI 导航，而在于偏好获取和干预校准，这暴露了胜任界面操作与成为值得信赖的个人助手之间存在的根本差距。

## Abstract
Personalized mobile agents that infer user preferences and calibrate proactive assistance hold great promise as everyday digital assistants, yet existing benchmarks fail to capture what this requires. Prior work evaluates preference recovery from static histories or intent prediction from fixed contexts. Neither tests whether an agent can elicit missing preferences through interaction, nor whether it can decide when to intervene, seek consent, or remain silent in a live GUI environment. We introduce KnowU-Bench, an online benchmark for personalized mobile agents built on a reproducible Android emulation environment, covering 42 general GUI tasks, 86 personalized tasks, and 64 proactive tasks. Unlike prior work that treats user preferences as static context, KnowU-Bench hides the user profile from the agent and exposes only behavioral logs, forcing genuine preference inference rather than context lookup. To support multi-turn preference elicitation, it instantiates an LLM-driven user simulator grounded in structured profiles, enabling realistic clarification dialogues and proactive consent handling. Beyond personalization, KnowU-Bench provides comprehensive evaluation of the complete proactive decision chain, including grounded GUI execution, consent negotiation, and post-rejection restraint, evaluated through a hybrid protocol combining rule-based verification with LLM-as-a-Judge scoring. Our experiments reveal a striking degradation: agents that excel at explicit task execution fall below 50% under vague instructions requiring user preference inference or intervention calibration, even for frontier models like Claude Sonnet 4.6. The core bottlenecks are not GUI navigation but preference acquisition and intervention calibration, exposing a fundamental gap between competent interface operation and trustworthy personal assistance.

*   **核心问题**：现有基准测试存在三大缺陷：
    1.  **偏好建模静态化**：通常将偏好视为静态上下文，而非通过交互动态获取。
    2.  **缺乏交互式评估**：无法测试智能体在信息缺失时是否会主动询问用户。
    3.  **主动性评估不完整**：未能覆盖“是否干预、是否征求许可、被拒绝后是否克制”的完整决策链。
*   **整体含义**：论文提出了 **KnowU-Bench**，旨在将评估重点从单纯的“指令跟随”转向“个性化理解”与“主动式服务”。

#### 2. 论文提出的方法论
*   **核心思想**：构建一个在线、可交互、可复现的 Android 模拟环境，通过“隐藏画像”强制智能体进行偏好推理。
*   **关键技术细节**：

    *   **信息不对称设计**：智能体只能访问用户的历史行为日志（Logs），必须从中推断偏好，或通过 `ask_user` 动作获取缺失信息。
    *   **混合评估协议**：结合**基于规则的检查**（验证最终状态，如订单是否生成）与 **LLM-as-a-Judge**（评估语义层面的偏好对齐、沟通语气及主动干预的合理性）。

#### 3. 实验设计
*   **任务场景**：共 192 个任务，分为三类：
    1.  **通用任务 (42个)**：基准 GUI 执行能力测试。
    2.  **个性化任务 (86个)**：指令模糊，需结合历史日志或询问用户来确定具体偏好。
    3.  **主动式任务 (64个)**：无显式指令，智能体需根据时间、地点和上下文决定是否介入。
*   **对比方法**：评估了 11 种主流模型，包括：

*   **Benchmark 设定**：在“全量日志+噪声”的复杂环境下进行在线执行评估。

#### 5. 实验数量与充分性
*   **实验规模**：对 11 个模型在 192 个任务上进行了系统评估，并针对 4 种不同的用户角色（研究员、开发者、学生、奶奶）进行了跨角色分析。

*   **评价**：实验设计较为充分，涵盖了从基础能力到高级决策的多个维度，且通过多角色和噪声干扰增强了实验的客观性与公平性。

#### 6. 主要结论与发现
*   **性能大幅下滑**：即使是顶尖模型（如 Claude 3.5 Sonnet），在通用任务上表现优异，但在涉及偏好推理和主动干预的任务中成功率跌破 50%。
*   **瓶颈转移**：当前的瓶颈已不再是 GUI 导航（点击、滑动等），而是**偏好获取**（不知道该问什么）和**干预校准**（不知道何时该保持沉默）。
*   **错误模式分析**：
    *   个性化任务中，66.7% 的失败源于“澄清不足”（未询问关键信息）。
    *   主动式任务中，80% 的失败源于“干预校准错误”（过度干预或错误被动）。

#### 7. 优点（亮点）
*   **真实性与可复现性**：基于真实的 Android 模拟器而非静态截图，且提供了完整的容器化方案。
*   **动态交互闭环**：引入用户模拟器，实现了“智能体询问-用户回答-智能体调整”的动态反馈链。
*   **评估维度全面**：不仅看任务是否完成，还评估了主动性决策中的“克制”能力（如用户拒绝后是否停止操作）。

#### 8. 不足与局限
*   **用户角色有限**：目前仅预设了 4 种角色，可能无法完全覆盖真实世界中极其多样化的用户行为模式。
*   **裁判偏见风险**：虽然验证了与人类的一致性，但 LLM-as-a-Judge 仍可能存在对特定模型语气的偏好或逻辑误判。
*   **应用限制**：主要集中在 Android 系统，未涉及跨设备（如 PC 或智能家居）的协同个性化评估。
*   **合成数据依赖**：历史日志和画像是合成生成的，虽然经过人工审核，但与真实用户的杂乱行为仍有差距。
