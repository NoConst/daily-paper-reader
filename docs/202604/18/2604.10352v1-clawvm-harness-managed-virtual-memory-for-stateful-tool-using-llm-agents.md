## 摘要
有状态的工具使用型 LLM 智能体将上下文窗口视为工作内存，然而现有的智能体框架（harnesses）对驻留性和持久性的管理仅停留在“尽力而为”的层面，导致了频繁发生的故障：压缩后的状态丢失、重置时的刷新绕过以及破坏性写回。我们提出了 ClawVM，这是一个虚拟内存层，它将状态管理为具有最小保真度不变式的有类型页，在 Token 预算下提供多分辨率表示，并在每个生命周期边界进行经验证的写回。由于框架已经负责组装提示词、协调工具并观察生命周期事件，因此它是天然的执行点；在此处建立契约使得驻留性和持久性变得确定且可审计。通过合成工作负载、12 个真实会话追踪以及对抗性压力测试，实验证明只要最小保真度集合符合 Token 预算（由离线真值确认），ClawVM 就能消除所有策略可控的故障，且每轮增加的策略引擎中值开销小于 50 微秒。

## Abstract
Stateful tool-using LLM agents treat the context window as working memory, yet today's agent harnesses manage residency and durability as best-effort, causing recurring failures: lost state after compaction, bypassed flushes on reset, and destructive writeback. We present \textsc{ClawVM}, a virtual memory layer that manages state as typed pages with minimum-fidelity invariants, multi-resolution representations under a token budget, and validated writeback at every lifecycle boundary. Because the harness already assembles prompts, mediates tools, and observes lifecycle events, it is the natural enforcement point; placing the contract there makes residency and durability deterministic and auditable. Across synthetic workloads, 12 real-session traces, and adversarial stress tests, \textsc{ClawVM} eliminates all policy-controllable faults whenever the minimum-fidelity set fits within the token budget, confirmed by an offline oracle, and adds median <50 microseconds of policy-engine overhead per turn.

#### 1. 核心问题与整体含义（研究动机和背景）
随着 LLM 智能体（如 OpenClaw、Claude Code）在长时会话中处理复杂任务（如邮件管理、代码编写），它们需要管理跨越数百个工具调用和多个会话的状态。目前的智能体框架（Harness）将上下文窗口视为“工作内存”，但其管理方式是“尽力而为”的（Best-effort）。这导致了三大类频繁故障：
*   **驻留性失败**：上下文压缩（Compaction）时丢失了关键指令或计划步骤。
*   **持久性失败**：在会话重置或保存时，未提交的状态被静默丢弃，或发生破坏性的覆盖写。
*   **可观测性失败**：检索失败时缺乏明确的错误代码，导致难以诊断。
论文的核心动机是借鉴操作系统的**虚拟内存（Virtual Memory）**机制，在框架层建立一套确定性的内存契约，确保关键状态的生存与持久化。

#### 2. 方法论：核心思想与关键技术
ClawVM 的核心思想是将智能体状态抽象为由框架强制管理的虚拟内存层。
*   **类型化页面（Typed Pages）**：将状态划分为 Bootstrap（引导/策略）、Constraint（约束）、Plan（计划）、Preference（偏好）、Evidence（证据）和 Conversation（对话片段）等类型。每种页面都有其最小保真度不变性（Invariants）。
*   **多分辨率表示（Multi-resolution Residency）**：页面可以以四种级别存在：Full（全文）、Compressed（压缩文本）、Structured（结构化字段）、Pointer（指针/句柄）。当 Token 预算紧张时，页面会根据预设路径降级（如从全文降级为结构化字段），而非直接丢弃。
*   **确定性选择策略**：采用两阶段算法。第一阶段安装所有硬固定（Hard-pinned）页面；第二阶段根据“边际效用/Token成本”进行贪心升级，直到耗尽预算。
*   **验证写回协议（Validated Writeback）**：采用三阶段事务机制（暂存、验证、提交）。框架会检查更新是否符合 Schema、是否存在破坏性覆盖，确保持久化过程非破坏性且可审计。
*   **故障模型**：定义了 Refetch（重新获取）、Bootstrap Fault（引导丢失）、Flush-miss（刷新遗漏）等可观测故障，使内存管理决策可回放、可诊断。
