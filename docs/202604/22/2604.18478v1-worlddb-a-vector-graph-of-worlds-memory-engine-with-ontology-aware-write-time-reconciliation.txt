# WorldDB: A Vector Graph-of-Worlds Memory Engine with Ontology-Aware Write-Time Reconciliation 

Harish Santhanalakshmi Ganesan ∗11Independent Researcher April 21, 2026 

Abstract 

Persistent memory is the bottleneck separating stateless language-model chatbots from long-running agentic systems. Retrieval-augmented generation (RAG) backed by flat vector stores fragments facts into chunks, loses cross-session identity, and has no first-class notion of supersession or contradiction. Recent bitemporal knowledge-graph systems (Graphiti, Memento, Hydra DB) address some of these failures with typed edges and valid-time metadata, but the graph itself remains flat—there is no mechanism for recursive composition, no content-addressed invariants on nodes, and edge types carry no behavior beyond a label. We present 

WorldDB , a memory engine organized around three non-standard commitments: (i) every node is a world —a container with its own interior subgraph, its own ontology scope, and its own composed embedding, recursive to arbitrary depth; (ii) nodes are content-addressed and immutable, so any edit produces a new hash at the edited node and every ancestor, giving a Merkle-style audit trail for free; (iii) edges are write-time programs—each edge type ships an 

on_insert /on_delete /on_query_rewrite handler that enforces its own semantics (supersession closes validity, contradicts preserves both sides, same_as stages a merge proposal), so no raw append path exists. On LongMemEval-s (500 questions, ∼ 115 k-token conversational stacks), WorldDB with Claude Opus 4.7 as answerer achieves 96.40% overall accuracy / 97.11% task-averaged , a +5 .61 pp improvement in overall accuracy over the previously reported Hydra DB state-of-the-art (90.79%) and +11 .20 pp over Supermemory (85.20%), with perfect scores on single-session-assistant recall and robust performance across temporal reasoning (96.24%), knowledge update (98.72%), and preference synthesis (96.67%). Ablations show that the engine’s graph layer—resolver-unified entities and typed refers_to edges—contributes 

+7 .0pp task-averaged independently of the underlying answerer. 

Keywords: agentic memory, knowledge graphs, content-addressing, bitemporal reasoning, long-term memory, LongMemEval.

More corrosive is context rot [7]—the gradual accumulation of stale, contradicted, or redundant content that crowds out salient signal even within a nominally sufficient context budget. Extending the window does not solve state persistence: every new session begins from a fresh slate with no model of the user’s evolving preferences or the superseded history of shared facts. Retrieval-augmented generation (RAG) [9] was intended to externalize memory, but the canonical implementation inherits three structural weaknesses that have proved difficult to patch: 

• Semantic fragmentation. Naive chunking destroys cross-segment dependencies. When an entity is introduced in one chunk and its attributes updated elsewhere, the system retrieves each fragment independently and produces “hallucinations of omission”—answers that miss half the evidence. 

• Temporal stagnation. Flat vector stores are chronology-agnostic. They cannot distinguish “I live in Austin now” from “I lived in Austin in 2022”; retrieval returns both and the language model invents a resolution. 

• Identity drift. Standard embeddings place “Sarah,” “the engineering lead,” and “my manager” in three nearby but distinct positions. Without an explicit resolver, these mentions never collapse into one entity—so questions that require tracking an entity across sessions silently fail. Recent bitemporal knowledge-graph systems (Graphiti/Zep [11], Memento [6], Hydra DB [12]) address subsets of these problems. Graphiti and Memento add valid-time metadata and entity resolution; Hydra DB adds a Git-style append-only temporal graph with sliding-window entity enrichment. All three report substantial gains on LongMemEval-s [16] over flat RAG—Hydra DB reaches 90.79% overall, Supermemory [15] 85.20% , Zep 71.2% , Mem0 [2] 29.07% . But the graph in each remains flat : nodes are labeled points, edges are typed pointers, and the typing carries no executable semantics. An edge (a, SUPERSEDES , b ) is a label; if the store is to treat b as historical, application code must do so explicitly on every query. We present WorldDB , a memory engine built on three non-standard commitments: 1. Recursive worlds. A node is not a row. A node is a world —a container with its own interior subgraph, its own ontology scope, its own composed embedding, and its own temporal extent. Worlds nest to arbitrary depth. A query inside world W cannot leak nodes from world W ′ unless an explicit refers_to edge is traversed. The edit propagation invariant (see §2.1) makes this structural; it is not enforced by application code. 2. Content-addressed immutability. Every node’s id is blake3 (τ ∥ n ∥ c ∥ sort (Cids ) ∥ sort (Eids ) ∥ tcreate ).A tiny edit to a leaf yields a new hash at the leaf and at every ancestor . This gives deduplication, verifiable references, and a Merkle-style audit trail without additional machinery. The blob is immutable by construction; validity intervals (when a fact is true) live on mutable columns of the node row so supersession can close them without violating content-addressing. 3. Edges as write-time programs. Each edge type in the default ontology ( contains , refers_to ,

The Contradicts handler records the conflict so default queries surface it; it does not delete either side. Users register custom edge types with custom handlers via a stable trait. 2Together these commitments define a memory substrate that is (a) structurally coherent—the graph topology is the answer, not an index over text—(b) verifiably auditable—every node is its own content hash—and (c) immune to the class of bugs where an append path is added “for performance” and silently violates the ontology. The never-appends rule (§2.2) is the thesis: no raw insertion path exists on edges; the single entry point is the reconciler.

Edit propagation invariant. If N ′̸ = N differs in content, children, edges, or created_at , then 

id (N )̸ = id (N ′). Consequently, editing a leaf L ∈ C(· · · C(N )) produces a new hash at L, at each intermediate ancestor, and at N . This makes the graph a Merkle tree over its own content: any node’s hash is a cryptographic witness to the full state of its interior. Dedup, verifiable cross-node references, and audit-trail replay come for free.

2.3 Bitemporality 

Every fact carries two orthogonal timestamps: tvalid (when the fact is true in the world) and tingested 

(when the system learned it). Queries can pin to either axis via an as-of parameter. Historical queries are cheap—the default view hides tvalid.to -closed nodes, but an explicit include-superseded 

flag returns them with their closed intervals intact. 

3 Reconciliation Pipeline 

No write reaches the store directly. Text (or structured input) flows through four stages: 1. Extract. Parse incoming text into candidate nodes and edges. The engine does not run an LLM itself (Appendix B covers why); the caller supplies an Extraction via the Extractor trait, which can be backed by any model. 2. Resolve. For each candidate node, run the tiered resolver of §3.1. The output is Resolved(m) ,

> Ambiguous(options)

, or New .3. Reconcile. Dispatch each candidate edge through its handler (§2.2). Handlers may close validity intervals, stage merge proposals, or flag contradictions—all atomically with the edge-row insert. 4. Commit. Content-addressed writes land; the ANN index receives embeddings incrementally. 

3.1 Tiered Resolver 

The resolver attempts five tiers in sequence and stops at the first confident match: 1. Exact. Case-insensitive name or alias equality via a node_name index. 2. Fuzzy. Jaro–Winkler similarity [8] ≥ 0.92 over name ∪ aliases, after FTS5-based blocking reduces the candidate pool from O(N ) to O(log N ) for large stores. 3. Phonetic. A Soundex-style key with PH →F, CK →K, KN →N, WR →R digraph rewrites so “Phillip” and “Filip” collide. 4. Embedding. Cosine similarity ≥ 0.88 via the HNSW index (§4.2). 5. Tiebreaker. When multiple tiers disagree on different hashes, a caller-supplied Tiebreaker 

(LLM, user, or programmatic) adjudicates. The default prefers the higher-confidence tier in the order exact > fuzzy > embedding > phonetic. Matches stage a same_as proposal in pending state. The engine never silently merges—confirmed merges require accept_merge(edge_hash) . Rejected proposals leave both sides fully independent.  

> equivalence_class(h)

follows accepted same_as edges transitively and returns the set of identity-class members; export_identity(h) unions each member’s provenance for GDPR-style auditing. 53.2 Incremental Reclustering 

After each ingest the resolver re-runs over the 2-hop neighborhood of the new nodes (Saeedi et al. [14]). This catches the case where a newly-written fact’s embedding has drifted into the orbit of an existing entity that wasn’t matched at the initial pass. Re-clustering is cheap because the frontier is bounded; it does not block the ingest commit.

4.2 HNSW Index 

Embeddings are stored in a separate node_embeddings(hash, dim, vec) table and mirrored into an in-memory HNSW index ( hnsw_rs , M = 16 , ef c = 200 , ef s = 64 ) built via parallel_insert_slice .On open, the index hydrates from SQL; on write, it updates incrementally. At 10 k nodes the HNSW path is 12 × faster than brute-force cosine in release mode ( 352 μs vs. 4.59 ms); at 100 k nodes the gap widens to 95 ×, P95 cosine dropping from 102 ms (brute force) to 1.07 ms (HNSW) on a 16 -dim index. A content/effective embedding split (§5) lets composed world embeddings overlay the effective vector without corrupting the stable content anchor used as the parent query in recompose passes—a feedback loop that silently degraded earlier versions of the composer.

6 Background Consolidator 

Over the lifetime of a long-running store, the graph grows in three predictable ways: (i) leaves accumulate, (ii) latent facts go unrecorded because no single turn stated them, (iii) contradictions emerge between distant regions that never interacted at write time. The consolidator is the engine’s “sleep cycle”—a background process that addresses all three: 

• Summary generation. Walk worlds older than a configurable min_age_secs , pull their children, and ask a pluggable Summarizer to produce one exhaustive-event summary node. Write World 

7→ Summary → details so queries with prefer_summaries() return the summary at the default depth and only descend on demand. 

• Inferred transitive edges. Compute one-pass closure on causes and subtype_of . New edges are flagged metadata.inferred=true and routed through the standard handler pipeline. 

• Contradiction sweep. Structural check: any node with multiple outgoing edges of a functional-relation type (user-registered) produces a flagged contradicts. At a 1000 -fact, 100 -topic scale the summary-preferred query runs 6.5× faster than the full-detail traversal ( 30 ms vs. 194 ms, median of 9 runs), and the detail remains reachable on demand—the “default-depth view is the summary” semantics are preserved without destroying the underlying leaves.

10 Conclusion 

WorldDB is a memory engine for agents built on three commitments that flat RAG and even recent bitemporal KG systems decline to make: recursive worlds, content-addressed immutability, and edges as write-time programs. Together they produce a substrate where the graph topology 

is the answer—facts are retrievable by structure as well as content, identity is first-class, and the append path is the reconciler. On LongMemEval-s the engine scores 96 .40% overall / 97 .11% 

task-averaged , a new state-of-the-art on this benchmark, with the largest gains concentrated in multi-session reasoning and temporal consistency. Ablations isolate an 11 pp task-averaged contribution to the engine’s graph layer, distinct from answerer capacity. Agentic systems will either develop persistent, queryable state or remain experimental. WorldDB 

is one concrete bet on what that state should look like.

Run once per session during ingest. The output is parsed with a relaxed JSON extractor that tolerates markdown fences Claude occasionally wraps around structured output.

A.3 Judge Prompt (Claude Sonnet 4.6) — LLM-as-Judge Protocol 

Each (question, reference, generated) triple is scored independently by a single Sonnet 4.6 call. We use a deliberately compact binary rubric ( CORRECT /WRONG ) with five explicit rules that trade off precision, paraphrase tolerance, and abstention handling. Unlike multi-dimensional scorers that ask the judge to emit JSON with is_correct , score , explanation , and key_matches fields [12], we rely on the judge-model’s single-token verdict. Two reasons: (i) richer output formats introduce more surface area for judge parsing errors to contaminate the metric; (ii) Claude Sonnet’s calibration on binary judgments is empirically tight, and secondary fields do not change the overall pass/fail. The 

max_tokens=8 cap forces the model to commit to a verdict in the first word or two. The full prompt: 

You are judging a memory-retrieval answer. Reply with exactly 

'CORRECT ' or 'WRONG ' and nothing else. 

14 Rules: - CORRECT if the generated answer contains (or clearly implies) the key fact from the reference. Extra accurate context is fine. - CORRECT if paraphrasing or minor wording differences express the same fact. - **If the reference itself says the information is missing, not enough, or unknown**, then a generated 'I don 't know ' or similar refusal IS CORRECT. A response that fabricates an answer when the reference says the info is missing is WRONG. - For numeric or named-value questions, the generated value must match the reference value (approximate match is fine if stated as such in the reference, e.g. 'about 15 '). - WRONG if the generated answer omits the key fact, states a different specific value, says 'I don 't know ' when the reference has a concrete answer, or contradicts the reference. Question: {question} Reference answer: {reference} Generated answer: {generated} Verdict: 

Verdict parsing. The judge’s response is strip().upper() -ed and matched by prefix: if it 

startswith("CORRECT") , the question counts as correct; anything else (including WRONG , partial tokens, refusals, or empty) counts as incorrect. This is strict-by-default — we never give the model credit for ambiguous output. 

Calibration tests. Three cases of the rubric are load-bearing and worth flagging: 1. Reference admits ignorance. When the LongMemEval reference itself says “The information provided is not enough...”, a generated "I don’t know" must score CORRECT. Early versions of our judge prompt (v8 in the ablation trace) marked this WRONG, dragging knowledge-update accuracy by ∼ 3pp. Rule three is the explicit fix. 2. Paraphrase tolerance. A reference answer of “GPS system not functioning correctly” is accepted against a generated “The first issue you had with your new car after its first service was with the GPS system on March 22.” 3. Numeric strictness. A reference “$185” is rejected against a generated “$65”; a reference “about 15 days” accepts “14 days” and “15 days.” Rule four makes the difference between an approximation judged present in the reference and an arbitrary rounding by the model. 

Why not an ensemble judge? We ran pilot comparisons with a GPT-4o judge in parallel on a 100-question subset. The two agreed on 94 % of verdicts; the 6% disagreements were almost entirely preference questions where subjective quality ratings diverge by judge (cf. the Hydra DB appendix which enumerates a separate preference-question rubric). We report Sonnet 4.6 as the single judge to keep the protocol reproducible and avoid ensemble-opacity in the metric. 

B Why No LLM in the Engine 

The engine never calls a language model on the read path. This reflects two observations: 15 1. LLM latency and cost are incompatible with query-path SLOs. A 200 ms answerer call multiplies by every hop of a multi-step query. At a 500 ms P95 target there is no budget for it. 2. LLM non-determinism corrupts the ontology. The reconciler’s guarantees rely on handler code being a function of input state; an LLM call breaks that, and the failure mode is silent. We weaken the constraint only at ingest time and consolidation time —both out-of-band, both auditable. The engine’s Extractor trait and Summarizer trait each accept a caller-supplied closure. Tests use deterministic stubs; production deployments wire in Anthropic / OpenAI / local models via a small bridge. 

C Reproducibility 

Every evaluation in §7 is driven by the prompts in Appendix A with temperature 0, pinned random seeds, and a single answerer and judge model per run. Ingestion is session-by-session; the ∼ 14 -minute wall-clock time on an 8-worker driver against the Claude and OpenAI endpoints listed in §7 is reproducible within a few-second variance attributable to API round-trip jitter. The synthetic retrieval study of §5 (50 worlds × 20 children, 100 held-out queries) and the 2 × 2,000 -operation reconciler fuzz in §7 use deterministic PRNG seeds; results do not drift between runs. 16
