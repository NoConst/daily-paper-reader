## 摘要
长期对话智能体必须决定将哪些对话轮次存储在外部内存中，然而目前的系统在每一轮对话中都依赖自回归大语言模型（LLM）生成来做出该决策。我们提出了 MemRouter，这是一种写端内存路由器，它将内存准入与下游回答主干网络解耦，并用基于嵌入的路由策略取代了逐轮的内存管理解码。MemRouter 将每一轮对话与近期上下文共同编码，通过冻结的 LLM 主干网络投影生成的嵌入，并使用轻量级分类头预测该轮次是否应被存储，仅需训练 1200 万个参数。在 LoCoMo 数据集上的受控匹配测试对比中，在检索流水线、回答提示词和问答主干网络（Qwen2.5-7B）保持一致的情况下，MemRouter 在所有问题类别上的表现均优于基于 LLM 的内存管理器（总 F1 分数为 52.0 对比 45.6，95% 置信区间不重叠），同时将内存管理的 p50 延迟从 970 毫秒降低至 58 毫秒。描述性因子平均进一步表明，学习到的准入机制相比随机存储使平均 F1 提升了 10.3，特定类别的提示词相比通用提示词提升了 5.2，而检索贡献了 0.7。这些结果表明，写端内存准入可以通过小型监督路由器来学习，而回答生成在长程对话问答中仍然是一个独立的下游组件。

## Abstract
Long-term conversational agents must decide which turns to store in external memory, yet recent systems rely on autoregressive LLM generation at every turn to make that decision. We present MemRouter, a write-side memory router that decouples memory admission from the downstream answer backbone and replaces per-turn memory-management decoding with an embedding-based routing policy. MemRouter encodes each turn together with recent context, projects the resulting embeddings through a frozen LLM backbone, and predicts whether the turn should be stored using lightweight classification heads while training only 12M parameters. Under a controlled matched-harness comparison on LoCoMo, where the retrieval pipeline, answer prompts, and QA backbone (Qwen2.5-7B) are held identical, MemRouter outperforms an LLM-based memory manager on every question category (overall F1 52.0 vs 45.6, non-overlapping 95% CIs) while reducing memory-management p50 latency from 970ms to 58ms. Descriptive factorial averaging further shows that learned admission improves mean F1 by +10.3 over random storage, category-specific prompting adds +5.2 over a generic prompt, and retrieval contributes +0.7. These results suggest that write-side memory admission can be learned by a small supervised router, while answer generation remains a separate downstream component in long-horizon conversational QA.

### 1. 核心问题与整体含义（研究动机和背景）
长期对话智能体（Long-term Conversational Agents）需要决定哪些对话信息值得存入外部记忆，以应对超长上下文。
*   **现有瓶颈**：目前的系统（如 Memory-R1, MemGPT）通常让大语言模型（LLM）在每一轮对话中通过自回归生成（Autoregressive Generation）来做出“存储/忽略”的决策。这导致了极高的推理延迟和计算成本（例如 600 轮对话需要 1200 次 LLM 调用），且内存管理策略往往与特定的模型主干深度耦合，难以迁移。
*   **核心目标**：提出一种高效、解耦的“写端（Write-side）”内存路由器，将内存准入决策从文本生成转变为嵌入空间的分类任务。

### 2. 方法论：核心思想与技术细节
MemRouter 的核心思想是**“内存即嵌入路由”**，通过冻结的 LLM 提取特征，仅训练极小的分类头。其流程分为三个阶段：
*   **上下文嵌入与投影（Stage 1）**：将当前对话轮次及其历史背景切分为固定大小的块（Chunks），使用预训练的 BGE 编码器生成嵌入向量。随后通过一个两层 MLP 投影层（约 8M 参数）将这些向量映射到 LLM 的隐藏层维度。
*   **骨干上下文建模（Stage 2）**：将投影后的嵌入序列输入到**冻结的** LLM 主干（如 Qwen2.5-7B）中。通过一次前向传播（Forward Pass）获取上下文相关的表征，避免了逐 token 的自回归解码。
*   **操作分类（Stage 3）**：利用轻量级分类头（约 4M 参数）预测两个任务：
    1.  **操作预测**：决定是存储（ADD）还是忽略（NOOP）。
    2.  **内容类型**：若存储，则分类为事实、情感、偏好、计划或常规。
*   **检索与生成**：在读取端，采用混合检索（语义余弦相似度 + BM25 关键词匹配），并根据问题类型（单跳、多跳、时间等）使用特定类别的提示词（Category-specific Prompts）驱动下游 LLM 生成答案。

*   **数据集**：主要使用 **LoCoMo**（长期对话记忆标准基准），包含多 session 对话，涉及单跳、多跳、时间推理、开放域等问题。

实验设计非常详尽且具有高度的客观性：

*   **性能卓越**：在相同条件下，MemRouter 的 F1 分数（52.0）显著优于基于 LLM 的管理器（45.6）。
*   **极速推理**：内存管理的 p50 延迟从 970ms 降至 **58ms**，提速约 **17 倍**；端到端吞吐量提升了 **10.8 倍**。
*   **解耦有效性**：证明了写端的准入决策可以独立于下游回答模型进行训练，且在更换回答主干（如换成 35B 模型）时，无需重新训练路由器即可提升性能。

### 7. 优点（亮点）
*   **高效率**：通过“前向传播分类”取代“自回归生成”，解决了长期对话系统的实时性痛点。
*   **轻量化**：仅需训练 12M 参数，对算力友好，且易于集成到现有系统中。
*   **架构优雅**：实现了写端（存储）与读端（回答）的完全解耦，符合模块化设计的原则。
*   **实验严谨**：通过 Matched-harness 实验排除了检索和提示词的干扰，真正孤立并验证了准入策略的优越性。

### 8. 不足与局限
*   **开放域推理较弱**：在开放域（Open-domain）问题上的表现不如 Memory-R1，这可能归因于冻结的回答主干在处理间接证据时的推理能力受限。

*   **应用限制**：主要针对文本对话，尚未扩展到多模态或更复杂的工具调用场景。
