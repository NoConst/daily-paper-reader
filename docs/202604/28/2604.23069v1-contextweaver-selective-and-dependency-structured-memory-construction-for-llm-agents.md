## 摘要
大语言模型（LLM）智能体在长上下文交互中经常面临挑战。随着智能体积累更多的交互历史，滑动窗口和提示词压缩等上下文管理方法可能会遗漏后续步骤所依赖的早期结构化信息。最近的基于检索的记忆系统虽然能提取相关内容，但仍忽视了多步推理所需的因果和逻辑结构。我们提出了 ContextWeaver，这是一个选择性且具有依赖结构化的记忆框架，它将智能体的交互轨迹组织成推理步骤图，并为未来的行动选择相关的上下文。与之前的上下文管理方法不同，ContextWeaver 支持：（1）基于依赖的构建与遍历，将每一步与其所依赖的早期步骤联系起来；（2）紧凑的依赖摘要，将从根节点到具体步骤的推理路径压缩为可重用的单元；以及（3）一个结合了执行反馈的轻量级验证层。在 SWE-Bench Verified 和 Lite 基准测试中，ContextWeaver 在 pass@1 指标上优于滑动窗口基准，同时减少了推理步骤和 Token 使用量。我们的观察表明，建模逻辑依赖关系为使用工具的 LLM 智能体提供了一种稳定且可扩展的记忆机制。

## Abstract
Large language model (LLM) agents often struggle in long-context interactions. As the agent accumulates more interaction history, context management approaches such as sliding window and prompt compression may omit earlier structured information that later steps rely on. Recent retrieval-based memory systems surface relevant content but still overlook the causal and logical structure needed for multi-step reasoning. We introduce ContextWeaver, a selective and dependency-structured memory framework that organizes an agent's interaction trace into a graph of reasoning steps and selects the relevant context for future actions. Unlike prior context management approaches, ContextWeaver supports: (1) dependency-based construction and traversal that link each step to the earlier steps it relies on; (2) compact dependency summarization that condenses root-to-step reasoning paths into reusable units; and (3) a lightweight validation layer that incorporates execution feedback. On the SWE-Bench Verified and Lite benchmarks, ContextWeaver improves performance over a sliding-window baseline in pass@1, while reducing reasoning steps and token usage. Our observations suggest that modeling logical dependencies provides a stable and scalable memory mechanism for LLM agents that use tools.

### 1. 核心问题与整体含义（研究动机和背景）
在长程交互任务（如软件工程、复杂规划）中，LLM 智能体产生的交互历史会迅速超过上下文窗口限制。现有的管理方法存在明显缺陷：
*   **滑动窗口（Sliding Window）：** 仅保留近期信息，容易丢失早期关键的决策依据。
*   **提示词压缩（Prompt Compression）：** 往往基于语义显著性，可能误删具有逻辑支撑作用的细节。
*   **基于检索的记忆（RAG）：** 依赖语义相似度，无法捕捉推理步骤之间的**因果（Causal）**和**逻辑（Logical）**依赖关系。

**研究动机：** 智能体的每一步行动通常依赖于之前的某个特定决策、工具输出或假设。如果这些依赖链条断裂，智能体会陷入重复探索、计划中断或产生逻辑矛盾。ContextWeaver 旨在通过构建“依赖结构化图记忆”来解决这一问题。

### 2. 方法论：核心思想与关键技术
ContextWeaver 将智能体的交互轨迹组织为一个**有向无环图（DAG）**，并根据逻辑依赖而非时间顺序来筛选上下文。

*   **核心思想：** 并非所有历史记录都同等重要，只有处于当前推理路径上的“祖先节点”才应被完整保留。
*   **关键技术组件：**
    1.  **节点提取（Node Extraction）：** 将每一步（Thought-Action-Observation）转化为结构化节点，包含摘要、依赖摘要和验证状态。
    2.  **父节点选择（Parent Selection）：** 使用 LLM 作为“逻辑依赖分析器”，识别当前步骤真正依赖的先前步骤（Top-m 个父节点），从而构建 DAG。
    3.  **祖先依赖构建（Ancestry Construction）：** 通过广度优先搜索（BFS）从当前节点向上遍历，提取所有直接或间接支撑当前步骤的祖先节点。
    4.  **上下文编织（Context Weaving）：** 在最终提示词中，完整保留祖先节点的内容；对于非祖先节点，仅保留 Thought 和 Action，并压缩/省略 Observation。
    5.  **验证与测试层（Validation Layer）：** 记录工具执行结果（如 pytest 输出）。失败的节点在后续构建中会被跳过，确保智能体基于“验证通过”的路径进行推理。

*   **稳定性：** 建模逻辑依赖能显著降低智能体在长轨迹任务中的“漂移”现象，使运行结果更具一致性（标准差更小）。
*   **模型差异：** 强模型（如 Claude 3.5 Sonnet）能更好地利用结构化记忆；而弱模型在生成摘要时可能遗漏关键细节，导致性能下降。

### 7. 优点：亮点与创新
*   **非线性记忆组织：** 突破了线性历史的限制，允许智能体在多个并行或交叉的推理分支中跳转。
*   **逻辑驱动而非语义驱动：** 解决了 RAG 无法处理因果逻辑的问题，确保上下文的“逻辑连贯性”。
*   **闭环验证：** 将执行反馈（测试结果）直接集成到记忆构建中，自动过滤无效路径。
*   **增量构建：** 依赖摘要是增量生成的，避免了重复处理全量历史带来的高昂成本。
